# Root Recovery Protocol

This file is only for recovery/resume scenarios; it does NOT replace the main flow `recursion-protocol.md`.
A normal new task still builds the tree via the main flow `init`.

## Trigger condition

When the user's message shows recovery intent (e.g. `恢复 / 接续 / recover / resume`) and there is no
`[ORCHESTRATION IDENTITY]` block, you are a recovery root.

The recovery root's primary goal is to find the most recent unfinished ultra-team run under the
current cwd, consume the intents and results of historical child agents, and continue on the existing site.

## Discover an unfinished run

First run:

```bash
python3 <skill>/scripts/agent_orchestrator.py recover-root \
  --cwd "$(pwd)" \
  --write-prompt /tmp/ultra-team-recovery-prompt.md
```

Behavior:

- Prefer the latest `running` run on the current cwd and current git branch.
- If the current branch has no unfinished run, report that it is not recoverable; if other branches under
  the same cwd have running runs, output those non-current-branch runs for reference, but do not advise
  switching or passing `--root-id`.
- If the user already provided a root_id, you may run:

```bash
python3 <skill>/scripts/agent_orchestrator.py recover-root \
  --cwd "$(pwd)" \
  --root-id <root_id> \
  --write-prompt /tmp/ultra-team-recovery-prompt.md
```

`recover-root` only reads state and generates a prompt; it never creates a run, never stops, and never
modifies business files.

## Resume rules

Once you have the prompt generated by `recover-root`, use that prompt as the recovery root's working
instruction and continue. The recovery prompt includes:

- The explicit `ultra team` trigger phrase.
- A recovery-scenario description.
- root_id, cwd, branch, original task.
- The historical agent tree expanded level by level from root.
- Each child agent's `intent`, kind, round, status, job_id, session_id, result, caveats.

The recovery root MUST:

- Not call `init`.
- First run `status --root-id <root_id> --json` to see the real current state.
- Consume historical child agents' intents and results to judge what is done, still running, failed, or lacking evidence.
- If direct children are still running, prefer continuing the corresponding round's `await-round`.
- If `await-round` returns `recover_idle_agent`, recover the idle background job per the returned command.
- If historical results are insufficient to prove completion, re-split the necessary review / implement / fix rounds for the current state.
- Not re-dispatch the same work already proven done by historical results.
- Still satisfy the main flow's review closeout gate before `finish`.

The recovery root only handles orchestration, state judgment, and a little wiring observation. As long as
child-agent history exists, business implementation, review, and fix MUST still go to
`kind=implement` / `kind=review` / `kind=fix` children. In particular, after a review returns
`needs_fix`, the recovery root MUST NOT modify business files directly to address the review's findings;
it MUST `dispatch --kind fix` or the necessary `dispatch --kind implement`, then `await-round`, then
re-`dispatch --kind review`.

The recovery root handles only its own direct children. Historical grandchildren or deeper stale running
states cannot be `stop`ed or cleaned by the root; to handle such nodes, return to their direct parent's
recovery context.

The body written to `dispatch --task` or `--task-stdin` may only contain business task details. Do NOT write the
`ultra team` trigger phrase, `[ORCHESTRATION IDENTITY]`, `[DISPATCH DISCIPLINE]`, or
agent_id/root_id placeholder blocks into the task body; the identity block and leaf discipline are
wrapped automatically by `dispatch`.

After the recovery root reaches a judgment, it MUST run the corresponding command at once. Do NOT just
narrate "should keep waiting / should review again / should dispatch fix" and then stop.

## Isolation from the main flow

This recovery protocol only describes on-site recovery; it does NOT change the semantics of the normal
`init` / `dispatch` / `await-round` / `finish` main flow.

A child agent in normal scheduling additionally records `intent`, for a readable summary in future
recovery prompts. This is only a state-record field; it does NOT change the child's execution prompt or
scheduling behavior.
